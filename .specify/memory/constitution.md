# Car V1 專案憲章

## 核心原則

### I. 程式碼品質卓越（不可妥協）

**Clean Code 標準：**
- 所有程式碼必須具備自我說明性，使用有意義的變數、函數和類別名稱
- 函數應遵循單一職責原則（SRP）— 一個函數只做一件事
- 函數最大長度：50 行；檔案最大長度：300 行
- 每個函數的循環複雜度不得超過 10
- 禁止程式碼重複 — 嚴格執行 DRY（Don't Repeat Yourself）原則

**程式碼組織：**
- 遵循一致的專案結構和架構模式
- 關注點分離：UI、業務邏輯、資料存取層必須隔離
- 使用依賴注入以提高可測試性和降低耦合度
- 所有公開 API 必須有完整的文件說明

**程式碼審查要求：**
- 所有程式碼變更在合併前必須經過同儕審查
- 審查者必須檢查：功能性、可讀性、測試覆蓋率、效能影響
- 生產環境程式碼禁止自我合併

---

### II. 測試標準（不可妥協）

**測試驅動開發：**
- TDD（測試驅動開發）對所有新功能為必須
- 工作流程：撰寫測試 → 驗證測試失敗 → 實作功能 → 驗證測試通過 → 重構
- 沒有對應測試的功能視為未完成

**測試覆蓋率要求：**
- 所有模組最低 80% 程式碼覆蓋率
- 關鍵路徑需要 100% 覆蓋率
- 合併前必須產生並審查覆蓋率報告

**測試金字塔：**
- **單元測試（70%）**：獨立測試個別函數和元件
- **整合測試（20%）**：測試元件互動和 API 契約
- **端對端測試（10%）**：測試關鍵使用者旅程和工作流程

**測試品質標準：**
- 測試必須獨立且可重複執行
- 使用描述性測試名稱：`should_[預期行為]_when_[條件]`
- 模擬外部依賴；單元測試中絕不連接真實外部服務
- 測試邊界案例、錯誤條件和邊界值

---

### III. 使用者體驗一致性

**設計系統合規：**
- 所有 UI 元件必須遵循既定的設計系統
- 使用一致的間距、字型排版和配色方案
- 元件必須可重用且可組合

**無障礙標準（A11y）：**
- WCAG 2.1 AA 合規為必須
- 所有互動元素必須支援鍵盤導航
- 提供適當的 ARIA 標籤和角色
- 維持最低對比度（一般文字 4.5:1）

**響應式設計：**
- 所有介面必須在定義的斷點間正常運作
- 新 UI 開發採用行動優先方法
- 觸控目標最小 44x44 像素

**使用者回饋：**
- 所有使用者操作必須提供即時視覺回饋
- 非同步操作需要載入狀態
- 錯誤訊息必須對使用者友善且可採取行動
- 破壞性或重要操作需要成功確認

**一致性規則：**
- 在應用程式中使用一致的術語
- 導航模式必須可預測
- 類似的操作應有類似的互動方式

---

### IV. 效能要求

**頁面載入效能：**
- 初始頁面載入：3G 連線下 < 3 秒
- 可互動時間（TTI）：< 5 秒
- 首次內容繪製（FCP）：< 1.5 秒
- 最大內容繪製（LCP）：< 2.5 秒

**執行時效能：**
- UI 互動必須在 100ms 內回應
- 動畫必須以 60fps 執行（每幀 16.67ms）
- API 回應時間：簡單查詢 < 200ms，複雜操作 < 1 秒

**資源最佳化：**
- 初始載入 Bundle 大小預算：gzip 後 < 200KB
- 圖片必須最佳化並採用延遲載入
- 大型模組實作程式碼分割

**記憶體管理：**
- 禁止記憶體洩漏 — 元件必須清理訂閱和監聽器
- 監控並限制長時間執行階段的記憶體使用
- 大型資料集實作分頁/虛擬化

**效能測試：**
- 關鍵路徑必須建立效能基準
- CI/CD 中進行自動化效能回歸測試
- Lighthouse 最低分數：效能 90+、無障礙 90+

## 開發工作流程

### 品質關卡

所有程式碼在合併前必須通過以下關卡：

1. **程式碼檢查（Linting）**：零錯誤、零警告
2. **型別檢查**：完整的型別安全（如適用）
3. **單元測試**：全部通過，覆蓋率門檻達標
4. **整合測試**：全部通過
5. **建置**：生產環境建置成功
6. **安全掃描**：無高危/嚴重漏洞
7. **效能**：相較基準無回歸

### 程式碼審查清單

- [ ] 程式碼遵循專案風格指南
- [ ] 測試完整且有意義
- [ ] 未引入不必要的複雜度
- [ ] 如需要已更新文件
- [ ] 已考慮效能影響
- [ ] 遵循安全最佳實踐
- [ ] 符合無障礙要求

---

## 安全標準

- 絕不提交機密資訊、API 金鑰或憑證
- 對所有使用者輸入進行驗證
- 輸出編碼以防止 XSS 攻擊
- 使用參數化查詢以防止 SQL 注入
- 實作適當的身份驗證和授權機制
- 定期更新依賴套件以修補安全漏洞

---

## 治理

本憲章優先於所有其他開發實踐和指南。

**執行：**
- 所有 Pull Request 必須驗證符合這些原則
- 不合規的程式碼將在審查時被拒絕
- 例外情況需要書面理由說明並經團隊批准

**修訂：**
- 修改本憲章需要：
  1. 附理由的書面提案
  2. 團隊審查和討論
  3. 多數批准
  4. 現有程式碼的遷移計劃（如適用）

**合規監控：**
- 自動化工具將執行可量測的要求
- 定期稽核主觀品質標準
- 每季審查憲章有效性
